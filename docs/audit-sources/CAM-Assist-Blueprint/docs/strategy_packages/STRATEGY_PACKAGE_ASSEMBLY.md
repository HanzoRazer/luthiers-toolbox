# Strategy Package Assembly

## Overview

The strategy package assembly CLI combines validated CAM Assist artifacts into a portable, reviewable package directory:

```
input: strategy.json (A2-validated)
  ↓
output: package_directory/
  ├── strategy.json      (copy of input)
  ├── review_packet.md   (A3-generated)
  └── manifest.json      (A4-generated)
```

## Non-Execution Declaration

Strategy packages **explicitly deny execution authority**. The assembly process:

1. Validates strategy against A2 contract
2. Rejects any execution authority claims
3. Generates review packet with non-execution notice
4. Creates manifest with required authority block

Any strategy with `execution_authority_claim: true` will fail assembly.

## Usage

### Basic Usage

```bash
python scripts/assemble_strategy_package.py strategy.json
```

Output: `./<strategy_stem>_package/`

### Custom Output Directory

```bash
python scripts/assemble_strategy_package.py strategy.json --out ./my_package
```

### Overwrite Existing Package

```bash
python scripts/assemble_strategy_package.py strategy.json --out ./my_package --force
```

## CLI Options

| Option | Description |
|--------|-------------|
| `strategy_json` | Path to the input strategy JSON file (required) |
| `--out <dir>` | Output directory for the package |
| `--force` | Overwrite existing output directory |

## Exit Codes

| Code | Meaning |
|------|---------|
| 0 | Package assembled successfully |
| 1 | Validation failure (strategy or manifest) |
| 2 | File/read/write error |

## Package Contents

Each assembled package contains:

### `strategy.json`

Copy of the input strategy, validated against the A2 schema contract.

### `review_packet.md`

Human-readable review document generated by A3, including:

- Non-execution notice
- Operation summary
- Material context
- Safety boundaries
- Coordinate frame
- Fret positions (for fret slot strategies)
- Operator sign-off checklist

### `manifest.json`

Package manifest with:

- Relative paths to bundled files
- Execution authority declarations
- Provenance metadata
- Version information

## Validation Pipeline

The assembler runs this validation sequence:

```
1. Load strategy JSON
2. Validate against A2 schema contract
   → Fail if invalid
   → Fail if execution_authority_claim: true
3. Generate review packet (A3)
4. Generate manifest (A4)
5. Write all files to output directory
6. Validate generated manifest
   → Fail if manifest invalid
7. Success
```

## Manifest Generation

The assembler generates manifests with these values:

| Field | Source |
|-------|--------|
| `manifest_version` | `"1.0.0"` |
| `package_type` | `"cam_assist_strategy_package"` |
| `operation_type` | Derived from strategy |
| `strategy_file` | `"strategy.json"` |
| `review_packet_file` | `"review_packet.md"` |
| `source_geometry_files` | `[]` (empty) |
| `created_at` | Current UTC timestamp |
| `cam_assist_version` | From version constant |

### Authority Block

Always generated with:

```json
{
  "non_execution_declaration": true,
  "execution_authority_claim": false,
  "requires_human_review": true
}
```

### Provenance

Generated with:

```json
{
  "source_spec_id": "<strategy_id or filename>",
  "created_by": "cam-assist-assembly",
  "derivation_notes": "Assembled from <filename> by CAM-A5 Strategy Package Assembly CLI."
}
```

## Operation Type Derivation

The `operation_type` for the manifest is derived in order:

1. `strategy.operation_type` (if present)
2. `strategy.operation_intent.operation_type` + `_strategy` suffix (if needed)
3. `strategy.operation` + `_strategy` (if operation is a string)
4. Fail if none found

## Overwrite Behavior

| Scenario | Without `--force` | With `--force` |
|----------|-------------------|----------------|
| Output doesn't exist | Create | Create |
| Output exists | Fail with error | Delete and recreate |

The `--force` option deletes the entire output directory before assembly. This ensures no stale files remain.

## Example

```bash
# Assemble from valid example
python scripts/assemble_strategy_package.py examples/valid/fret_slot_strategy.json \
  --out examples/packages/fret_slot_strategy_example

# Validate the generated manifest
python scripts/validate_manifest.py examples/packages/fret_slot_strategy_example/manifest.json
```

## Integration with A1-A4

| Component | Role in Assembly |
|-----------|------------------|
| A1 | Schema foundation |
| A2 | Strategy validation |
| A3 | Review packet generation |
| A4 | Manifest validation |
| A5 | Assembly orchestration |

## What This Is Not

The assembly CLI does **not**:

- Generate G-code
- Authorize machine execution
- Replace CAM software
- Bypass human review requirements
- Create toolpaths

It produces **documentation packages** for human review and downstream CAM tooling.

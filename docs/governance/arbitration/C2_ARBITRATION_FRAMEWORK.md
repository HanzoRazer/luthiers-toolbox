## 1. Authority Statement

This document establishes the constitutional arbitration framework for C2 federalism.

This framework:
- Defines arbitration packet structure
- Establishes decomposition-first discipline
- Documents escalation boundaries
- Preserves dissent visibility
- Creates federation gates

This framework does NOT:
- Resolve semantic disputes
- Assign canonical ownership
- Mandate implementation
- Create CI enforcement


## 4. Decomposition-First Rule

### Rule Statement

```
No semantic surface may be federated until decomposition is complete.
Decomposition documents what exists.
Decomposition does not choose winners.
```

### Decomposition Requirements

| Requirement | Purpose |
|-------------|---------|
| Layer identification | Map semantic layers involved |
| Authority candidate enumeration | List all ownership claimants |
| Propagation path mapping | Document value flow across boundaries |
| Collision classification | Type and severity of semantic overlap |
| Dependency topology | Systems affected by resolution |

### Decomposition Outputs

| Output | Description |
|--------|-------------|
| Semantic layer map | Which layers contain the disputed term |
| Candidate matrix | Who claims authority and by what evidence |
| Propagation diagram | How values move across system boundaries |
| Collision type | Overload, divergence, gap, or escalation |
| Affected domain list | Systems requiring notification |

## 5. Arbitration Packet Structure

### Required Sections (15)

Every arbitration packet MUST contain these sections:

| # | Section | Purpose |
|---|---------|---------|
| 1 | **Packet ID** | Unique identifier (format: `C2_{DOMAIN}_ARBITRATION_PACKET_{NNN}`) |
| 2 | **Semantic Surface** | The term, concept, or boundary under arbitration |
| 3 | **Frozen C1 References** | Specific C1 evidence (inventories, collisions, findings) |
| 4 | **Collision References** | COLL-* IDs from collision logs |
| 5 | **Authority Claimants** | Systems currently asserting authority |
| 6 | **Ownership Candidates** | Potential canonical owners with evidence and gaps |
| 7 | **Dependency Topology** | Domains affected directly and indirectly |
| 8 | **Propagation Analysis** | How values flow, where escalation risks exist |
| 9 | **Constitutional Risks** | Invariant violations, layer collapse, authority drift |
| 10 | **Decomposition Proposal** | Layer boundaries, namespace strategy |
| 11 | **Federation Blockers** | What must resolve before federation |
| 12 | **Unresolved Tensions** | Disagreements that remain after decomposition |
| 13 | **Dissent Visibility** | Documented disagreements, deferred questions |
| 14 | **Escalation Triggers** | Conditions requiring human arbitration |
| 15 | **Ratification Prerequisites** | What must be true before ratification vote |

### Packet States

| State | Meaning | Transitions |
|-------|---------|-------------|
| `DRAFT` | Initial decomposition in progress | → DECOMPOSITION_COMPLETE |
| `DECOMPOSITION_COMPLETE` | All 15 sections documented | → TERMINAL_REVIEW |
| `TERMINAL_REVIEW` | Awaiting terminal sign-off | → ARBITRATION_READY |
| `ARBITRATION_READY` | Ready for human decision | → RATIFIED or DEFERRED |
| `RATIFIED` | Authority assigned, federation approved | → ENFORCEMENT |
| `DEFERRED` | Insufficient consensus, parked | → DRAFT (reopen) |
| `ENFORCEMENT` | CI/runtime enforcement active | Terminal |

## 6. Escalation Model

### Escalation Tiers

| Tier | Trigger | Handler |
|------|---------|---------|
| **Tier 0** | Decomposition question | Terminal owner |
| **Tier 1** | Cross-terminal dependency | Terminal 1 coordination |
| **Tier 2** | Authority ambiguity | Arbitration packet |
| **Tier 3** | Constitutional invariant at risk | Human arbitration required |
| **Tier 4** | Governance model inadequacy | C3 deferral |

### Escalation Triggers

| Trigger | Description | Tier |
|---------|-------------|------|
| Single-term multiple authorities | Same term claimed by >1 system | 2 |
| Layer boundary unclear | Ambiguous semantic layer | 1 |
| Propagation path uncontrolled | Values escaping containment | 2 |
| Invariant violation risk | C0 invariant at risk | 3 |
| Sandbox escape detected | Pre-governance vocabulary in production | 2 |
| Authority origin unresolved | No candidate has clear evidence | 3 |
| Terminal disagreement | Terminals cannot reach consensus | 3 |
| Governance gap | Framework cannot express the problem | 4 |

### Escalation Protocol

```
1. Document the trigger condition
2. Reference the C1 evidence
3. Identify affected packets
4. Notify affected terminals
5. Update packet state
6. If Tier 3+: flag for human arbitration
7. If Tier 4: create C3 deferral ticket
```

## 7. Dissent Preservation

### Dissent Principle

```
Dissent is signal, not noise.
Unresolved disagreement must remain visible.
Forced consensus creates hidden governance debt.
```

### Dissent Requirements

| Requirement | Implementation |
|-------------|----------------|
| Visibility | All dissent recorded in packet Section 13 |
| Attribution | Dissenting terminal/domain identified |
| Reasoning | Why the dissent exists, what evidence supports it |
| Deferred questions | Questions parked, not deleted |
| Reopening path | How dissent can trigger packet revision |

### Dissent Categories

| Category | Meaning | Resolution Path |
|----------|---------|-----------------|
| Technical disagreement | Different technical analysis | Additional evidence |
| Authority disagreement | Disputed ownership claims | Human arbitration |
| Scope disagreement | What should be in/out of packet | Terminal coordination |
| Timing disagreement | When resolution should occur | Sequencing adjustment |
| Framework disagreement | Governance model inadequacy | C3 deferral |

## 8. Federation Gates

### Gate Structure

Federation requires passing through sequential gates:

```
GATE 1: Decomposition Complete
        └── All 15 packet sections documented
        └── C1 evidence frozen and referenced
        └── Collision IDs assigned

GATE 2: Terminal Review Complete
        └── All affected terminals have reviewed
        └── Dissent documented
        └── No Tier 3+ escalations unresolved

GATE 3: Arbitration Ready
        └── Authority candidates enumerated
        └── No constitutional invariant violations
        └── Federation blockers addressed

GATE 4: Ratification Vote
        └── Human approval obtained
        └── Authority assignment documented
        └── Implementation path clear

GATE 5: Federation Active
        └── Authority registered in 7M/7N
        └── Consumer contracts established
        └── Enforcement enabled
```

### Gate Bypass Prohibition

```
INVARIANT: No gate may be bypassed.
INVARIANT: Gate ordering is constitutional.
INVARIANT: Failed gates must be addressed, not ignored.
```

## 9. Arbitration Constraints

### What Arbitration MAY Do

| Action | Scope |
|--------|-------|
| Decompose semantic surfaces | Document layers, candidates, paths |
| Enumerate authority candidates | List claimants with evidence |
| Map propagation paths | Document value flow |
| Identify constitutional risks | Flag invariant violations |
| Preserve dissent | Record disagreements visibly |
| Recommend resolution | Propose, not mandate |

### What Arbitration MUST NOT Do

| Prohibition | Reason |
|-------------|--------|
| Assign authority without ratification | Authority requires human decision |
| Mandate implementation | Implementation follows ratification |
| Create CI enforcement | Enforcement is C3 phase |
| Suppress dissent | Dissent is constitutional signal |
| Bypass gates | Gate ordering is invariant |
| Resolve without evidence | C1 evidence is constitutional foundation |

### Constitutional Boundary

```
Arbitration prepares decisions.
Arbitration does not make decisions.
Human judgment assigns authority.
Human judgment ratifies federation.
```

## 10. Packet Lifecycle

### Creation

```
1. Identify semantic surface requiring arbitration
2. Reference C1 evidence (inventories, collisions, findings)
3. Create packet with DRAFT state
4. Assign packet ID
5. Notify affected terminals
```

### Decomposition

```
1. Complete all 15 required sections
2. Map authority candidates
3. Document propagation paths
4. Identify constitutional risks
5. Propose layer boundaries
6. Transition to DECOMPOSITION_COMPLETE
```

### Review

```
1. Each affected terminal reviews
2. Terminal findings documented
3. Dissent recorded
4. Escalations identified
5. Transition to TERMINAL_REVIEW then ARBITRATION_READY
```

### Resolution

```
1. Human reviews packet
2. Authority decision made
3. Ratification recorded
4. Implementation path documented
5. Transition to RATIFIED
```

### Deferral

```
1. Insufficient consensus identified
2. Deferral reason documented
3. Reopening conditions specified
4. Transition to DEFERRED
5. Packet remains visible (not deleted)
```

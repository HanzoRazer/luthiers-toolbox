# SECTION 3 — CANONICAL IMPLEMENTATION RULES

## Rule 1 — One Subsystem Owns One Kind of Truth

| Subsystem | Owns |
|-----------|------|
| Geometry | Geometry truth |
| Runtime | Execution planning |
| Validation | Validation verdicts |
| Translators | Serialization |
| Archives | Observational preservation |

**Enforcement:** Never merge authority domains casually.

---

## Rule 2 — Contracts Are Ontology

Contracts are not DTOs.

Contracts are:

```text
domain vocabulary
```

This means contracts define:

- semantic truth
- lifecycle semantics
- authority boundaries
- permitted meaning

**Contract Creation Checklist:**

Every new contract must answer:

1. What truth does this contract own?
2. What truth must it never own?

---

## Rule 3 — Intent Is Separate from Execution

This applies repo-wide.

| Domain | Intent | Execution |
|--------|--------|-----------|
| CAM | Design intent | Runtime machining |
| Acoustics | Estimate | Measurement → Calibrated prediction |
| Workspace | Experimental | Canonical |

**Enforcement:** This separation is a canonical architectural invariant.

---

## Rule 4 — Runtime Cannot Redefine Ontology

- emit governed artifacts

### Runtimes May Not:

# SECTION 5 — CANONICAL FUTURE DIRECTION: CAM

## 5.1 Prohibited CAM Patterns

The repository must stop building:

- isolated generators
- direct G-code emitters
- operation-specific runtime islands

## 5.2 Required CAM Architecture

The future CAM direction is:

```text
Design Tool
    ↓
Intent Contract
    ↓
Runtime Dispatcher
    ↓
Operation Runtime Plugin
    ↓
Validation
    ↓
Preview / Simulation
    ↓
Machine Adapter
    ↓
Export
```

The runtime dispatcher is the missing architectural center.

---

## 5.3 CAM Implementation Sequence

### Step 1 — Typed Intent Contracts

Future CAM contracts should evolve into:

| Contract | Purpose |
|----------|---------|
| CamIntentV1 | Base intent envelope |
| RosetteDesignV1 | Rosette-specific intent |
| EngravingDesignV1 | Engraving-specific intent |
| BindingDesignV1 | Binding-specific intent |
| SurfaceTargetV1 | Surface targeting |
| ToolRequirementV1 | Tooling requirements |
| OperationManifestV1 | Operation manifest |
| ValidationReportV1 | Validation results |
| RuntimeArtifactV1 | Runtime artifacts |

The envelope remains stable. Subcontracts specialize by operation.

---

### Step 2 — Runtime Dispatcher

Future CAM implementation should prioritize:

```text
RuntimeDispatcher
```

before adding additional machining features.

**Dispatcher Responsibilities:**

```text
resolve intent
→ resolve runtime
→ query geometry
→ validate environment
→ generate strategy
→ preview
→ export
```

---

### Step 3 — Runtime Interface

All runtimes should converge toward:

```python
class CamOperationRuntime:
    def validate(self) -> ValidationResult: ...
    def resolve_geometry(self) -> GeometryResult: ...
    def generate_strategy(self) -> StrategyResult: ...
    def generate_preview(self) -> PreviewResult: ...
    def export(self) -> ExportResult: ...
```

This standardization matters more than adding new operations.

---

### Step 4 — Geometry Query Service

Surface-aware machining requires governed geometry queries:

| Query | Purpose |
|-------|---------|
| `query_surface_normal()` | Surface normal at point |
| `query_local_curvature()` | Curvature analysis |
| `query_thickness()` | Material thickness |
| `query_keepout_zones()` | Restricted regions |
| `query_fixture_transform()` | Fixture positioning |

This becomes canonical infrastructure.

---

### Step 5 — Adaptive Intelligence Later

Only after:

- deterministic contracts
- provenance
- authority
- runtime stability

should adaptive machining intelligence be introduced.

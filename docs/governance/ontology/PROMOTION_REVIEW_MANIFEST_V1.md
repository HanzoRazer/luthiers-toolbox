## Purpose

This document defines the governance contract for promotion review — the process by which instrument evidence transitions from staging (Tier 3) to curated candidate (Tier 2).

**Core Rule:**

```
PromotionReviewManifestV1 records review.
It does not create truth.
```

**This document:**
- Defines what promotion review means
- Establishes reviewer authority boundaries
- Specifies provenance requirements
- Defines arbitration record structure
- Establishes rejection pathways

**This document does NOT:**
- Create canonical instrument truth (Tier 1)
- Implement executable schemas
- Migrate existing data
- Authorize runtime consumption of curated data

---

## Authority Statement

Promotion review is:

```
semantic arbitration preparation
```

NOT:

```
canonical ratification
```

A reviewed record is a curated candidate, not canonical truth. The distinction matters because:

- Curated candidates may still contain unresolved semantic ambiguity
- Curated candidates require further governance approval for Tier 1 promotion
- Curated candidates are not consumable by production runtimes

## Promotion Tier Model

### Tier 3: Evidence (Non-Canonical)

```
morphology_harvest/outputs/
vectorizer extractions
AI vision outputs
blueprint dimension extractions
```

**Characteristics:**
- Generated by extraction systems
- No human review
- May contain errors, conflicts, ambiguity
- NOT consumable by any runtime
- Gitignored / ephemeral

### Tier 2: Curated Candidate (Reviewed)

```
data_registry/curated/instruments/
```

**Characteristics:**
- Human-reviewed evidence
- Provenance documented
- Semantic conflicts flagged
- Ontology field mapping attempted
- NOT YET canonical
- May be rejected or revised

### Tier 1: Canonical (Governed)

```
data_registry/system/instruments/
```

**Characteristics:**
- Governance-ratified
- Ontology-compliant
- Provenance complete
- Consumable by runtimes
- Version-controlled

**This manifest governs Tier 3 → Tier 2 only.**

Tier 2 → Tier 1 promotion requires separate governance approval and is OUT OF SCOPE.

## What Promotion Review Is

Promotion review is the process of:

1. **Examining** evidence from Tier 3 staging
2. **Verifying** extraction quality and plausibility
3. **Mapping** extracted values to ontology fields
4. **Documenting** provenance and confidence
5. **Flagging** unresolved semantic conflicts
6. **Recording** the review decision

Promotion review is NOT:

- Automatic normalization
- Silent conflict resolution
- Canonical truth declaration
- Runtime authorization

---

## Reviewer Authority

### Reviewer May

| Action | Description |
|--------|-------------|
| APPROVE | Accept evidence as curated candidate |
| APPROVE_WITH_NOTES | Accept with documented concerns |
| REQUEST_CLARIFICATION | Return for additional information |
| REJECT | Decline promotion with documented reason |
| DEFER | Postpone decision pending governance clarification |

### Reviewer May NOT

| Action | Reason |
|--------|--------|
| Declare canonical truth | Requires Tier 1 governance |
| Resolve semantic conflicts silently | Conflicts must be documented |
| Modify source evidence | Tier 3 is read-only during review |
| Authorize runtime consumption | Tier 2 is not consumable |
| Skip provenance documentation | Provenance is mandatory |

## Provenance Requirements

Every promoted record MUST include:

### Source Provenance

| Field | Required | Description |
|-------|----------|-------------|
| `source_system` | YES | Extraction system that produced evidence |
| `source_path` | YES | Original file/document path |
| `source_timestamp` | YES | When evidence was extracted |
| `extraction_method` | YES | How values were extracted |
| `extraction_confidence` | YES | System-reported confidence (0.0-1.0) |

### Review Provenance

| Field | Required | Description |
|-------|----------|-------------|
| `reviewer_id` | YES | Who performed the review |
| `review_timestamp` | YES | When review occurred |
| `review_decision` | YES | APPROVE / APPROVE_WITH_NOTES / REJECT / DEFER |
| `review_confidence` | YES | Reviewer confidence (0.0-1.0) |
| `review_notes` | CONDITIONAL | Required if decision is not plain APPROVE |

### Ontology Mapping Provenance

| Field | Required | Description |
|-------|----------|-------------|
| `ontology_version` | YES | Which ontology version was used for mapping |
| `field_mappings` | YES | Source field → ontology field mappings |
| `unmapped_fields` | YES | Fields that could not be mapped |
| `mapping_confidence` | YES | Confidence in mappings (0.0-1.0) |

## Arbitration Record Structure

When semantic conflicts are identified, they MUST be recorded:

### Conflict Record

```
conflict_id: unique identifier
conflict_type: DIMENSIONAL | TERMINOLOGY | CONVENTION | DOMAIN
affected_fields: [list of ontology fields]
observed_values: [list of conflicting values with sources]
likely_cause: description of why conflict exists
ontology_implication: how this affects semantic meaning
resolution_status: UNRESOLVED | DEFERRED | PROPOSED_RESOLUTION
proposed_resolution: (if status is PROPOSED_RESOLUTION)
resolution_rationale: (if status is PROPOSED_RESOLUTION)
```

### Conflict Types

| Type | Description |
|------|-------------|
| DIMENSIONAL | Same field, different numeric values |
| TERMINOLOGY | Different field names for same concept |
| CONVENTION | Different measurement conventions applied |
| DOMAIN | Unclear which ontology domain owns the field |

---

## Normalization Justification

If any normalization is applied during review, it MUST be justified:

### Normalization Record

```
normalization_id: unique identifier
source_field: original field name
source_value: original value
target_field: ontology field name
target_value: normalized value (if different)
normalization_type: RENAME | CONVERT | DERIVE | NONE
justification: why this normalization is appropriate
confidence: confidence in normalization (0.0-1.0)
reversible: whether original can be recovered
```

### Normalization Types

| Type | Description | Example |
|------|-------------|---------|
| RENAME | Field name change only | `body_length` → `body_length_mm` |
| CONVERT | Unit conversion | inches → mm |
| DERIVE | Value derived from other fields | bbox height from outline |
| NONE | No normalization needed | Direct mapping |

### Forbidden Normalizations

| Normalization | Reason |
|---------------|--------|
| bbox → physical (implicit) | Conflates measurement conventions |
| Averaging conflicting values | Creates meaningless hybrid |
| Dropping conflicting sources | Loses provenance |
| Silent type coercion | May lose precision |

## Confidence Scoring

### Extraction Confidence (System-Reported)

| Range | Meaning |
|-------|---------|
| 0.9 - 1.0 | High confidence extraction |
| 0.7 - 0.9 | Moderate confidence, likely correct |
| 0.5 - 0.7 | Low confidence, requires verification |
| 0.0 - 0.5 | Very low confidence, likely needs re-extraction |

### Review Confidence (Reviewer-Assigned)

| Range | Meaning |
|-------|---------|
| 0.9 - 1.0 | Reviewer highly confident in mapping |
| 0.7 - 0.9 | Reviewer moderately confident |
| 0.5 - 0.7 | Reviewer uncertain, flagging concerns |
| 0.0 - 0.5 | Reviewer recommends rejection or deferral |

### Combined Confidence

```
combined_confidence = extraction_confidence × review_confidence
```

Records with `combined_confidence < 0.5` SHOULD be flagged for additional review before Tier 1 consideration.

## Rejection Pathways

### Rejection Reasons

| Reason Code | Description |
|-------------|-------------|
| `EXTRACTION_QUALITY` | Source extraction is too low quality |
| `UNMAPPABLE_FIELDS` | Critical fields cannot map to ontology |
| `UNRESOLVABLE_CONFLICT` | Semantic conflict cannot be documented |
| `PROVENANCE_INCOMPLETE` | Required provenance information missing |
| `DOMAIN_MISMATCH` | Evidence belongs to different ontology domain |
| `DUPLICATE_EVIDENCE` | Equivalent record already in Tier 2 |

### Rejection Record

```
rejection_id: unique identifier
source_record_id: which evidence was rejected
rejection_reason: reason code
rejection_details: human-readable explanation
rejection_timestamp: when rejection occurred
rejector_id: who rejected
remediation_path: (optional) how evidence could be fixed
```

### Post-Rejection Options

| Option | Description |
|--------|-------------|
| Re-extract | Return to source, extract again with different method |
| Manual entry | Human creates evidence from original source |
| Abandon | Source cannot provide usable evidence |
| Escalate | Governance decision required |

---

## Deferral Pathways

### Deferral Reasons

| Reason Code | Description |
|-------------|-------------|
| `ONTOLOGY_GAP` | Required ontology field does not exist yet |
| `GOVERNANCE_PENDING` | Waiting for governance decision |
| `CONFLICT_ARBITRATION` | Semantic conflict requires governance input |
| `DOMAIN_BOUNDARY` | Unclear which domain should own this data |

### Deferral Record

```
deferral_id: unique identifier
source_record_id: which evidence was deferred
deferral_reason: reason code
deferral_details: human-readable explanation
deferral_timestamp: when deferral occurred
deferrer_id: who deferred
unblock_condition: what must happen to unblock
```

  - rejection_record: {...}

  - deferral_record: {...}

  - description of items requiring governance attention

## What This Manifest Does NOT Do

### Does NOT Create Canonical Truth

Curated candidates (Tier 2) are not canonical. They are:

```
reviewed evidence awaiting governance ratification
```

Production runtimes MUST NOT consume Tier 2 data.

### Does NOT Resolve Semantic Conflicts

Conflicts are documented, not resolved. Resolution requires:

```
governance arbitration at Tier 1 promotion
```

### Does NOT Authorize Migration

This manifest records review decisions. It does not:

- Modify existing canonical sources
- Rewrite consumers
- Delete Tier 3 staging
- Merge into production storage

### Does NOT Implement Schema

This is a governance document, not executable code. Executable schema implementation requires:

```
ontology ratification first
```

# ADR-0012: Epistemic Status Taxonomy

**Status:** Accepted  
**Date:** 2026-05-24  
**Supersedes:** None  
**Extends:** ADR-0011 (Measurement Authority Constitutional Definition)  
**Scope:** Constitutional doctrine (no runtime implementation)

---

## Context

ADR-0011 established that measurement authority does not transfer to derived artifacts, and that epistemic status must be tracked to prevent authority laundering.

This ADR defines the constitutional taxonomy of epistemic states. All artifacts, fields, and outputs in the tap-tone-pi ecosystem must be classifiable into one of these states.

The taxonomy is designed to be:
- **Stable** — primary categories should not change frequently
- **Exhaustive** — every artifact type must fit one category
- **Non-hierarchical** — categories are distinct, not nested
- **Implementation-neutral** — no runtime mechanics defined here

---

## Decision

### 1. Primary Epistemic Status Categories

| Status | Definition | Authority Level |
|--------|------------|-----------------|
| **OBSERVED** | Directly captured from physical sensor | Measurement-authoritative |
| **DERIVED** | Mathematical transform of observed data | Computationally authoritative |
| **ESTIMATED** | Inferential approximation from observed/derived data | Approximation only |
| **PREDICTED** | Model-generated future or inferred state | Model-dependent |
| **HEURISTIC** | Advisory/non-authoritative guidance | No authority |
| **OPERATOR-ANNOTATED** | Human judgment or annotation | Operator authority only |
| **EXTERNALLY-SOURCED** | Imported from outside the system | External authority |

### 2. Category Definitions

#### OBSERVED

**Definition:** Data captured directly from physical sensors during acquisition.

**Examples:**
- Raw audio waveform from microphone
- Timestamp of capture event
- Device identification at capture time
- Environmental readings during capture

**Authority:** Measurement-authoritative as defined in ADR-0011.

**Constraints:**
- Must pass through canonical capture pipeline
- Must have provenance metadata
- Cannot be modified post-capture

---

#### DERIVED

**Definition:** Data produced by deterministic mathematical transformation of OBSERVED data.

**Examples:**
- FFT magnitude spectrum
- Transfer function H(f)
- Coherence γ²(f)
- RMS amplitude
- Peak frequency detection

**Authority:** Computationally authoritative — given the same input, produces the same output.

**Constraints:**
- Transform must be deterministic and reproducible
- Source OBSERVED data must be identified
- Does NOT inherit measurement authority

**Key distinction:** DERIVED data is "correct" in the computational sense, but correctness does not imply truth about the physical system.

---

#### ESTIMATED

**Definition:** Data produced by inferential approximation, where the result depends on assumptions or models.

**Examples:**
- Wood density estimate from frequency and dimensions
- Radiation coefficient approximation
- Mode shape interpolation from sparse grid
- Damping factor extraction

**Authority:** Approximation only — depends on model validity.

**Constraints:**
- Assumptions must be documentable
- Uncertainty bounds should be expressible
- Does NOT inherit authority from source data

**Key distinction:** ESTIMATED data involves inference beyond direct computation.

---

#### PREDICTED

**Definition:** Data generated by models to represent future states, counterfactuals, or inferred conditions.

**Examples:**
- Wolf tone severity after hypothetical mass addition
- Frequency shift prediction from thickness change
- Expected coherence improvement from averaging

**Authority:** Model-dependent — validity depends entirely on model accuracy.

**Constraints:**
- Model must be identified
- Predictions must not be presented as measurements
- Does NOT inherit any upstream authority

**Key distinction:** PREDICTED data describes what might be, not what was observed.

---

#### HEURISTIC

**Definition:** Advisory output that provides guidance without claiming authority.

**Examples:**
- AGE attention directives
- Wolf mitigation suggestions
- Quality grade classifications (A/B/C/D)
- "Suggested for review" flags

**Authority:** None — explicitly non-authoritative.

**Constraints:**
- Must use allowed advisory vocabulary (per ADR-0010)
- Must not be presented as measurement or validation
- Operator may ignore without consequence

**Key distinction:** HEURISTIC data is constitutionally forbidden from acquiring authority through any mechanism.

---

#### OPERATOR-ANNOTATED

**Definition:** Data entered, modified, or approved by human operator judgment.

**Examples:**
- Specimen identification labels
- Operator notes and comments
- Manual corrections to detected values
- Acceptance/rejection decisions
- Review flags (reviewed, flagged, needs-followup)

**Authority:** Operator authority only — represents human judgment, not system validation.

**Constraints:**
- Operator identity should be traceable
- Does not validate or canonize system outputs
- Cannot be overridden by system confidence

**Key distinction:** OPERATOR-ANNOTATED data is constitutionally protected by operator sovereignty (ADR-0010, ADR-0011).

---

#### EXTERNALLY-SOURCED

**Definition:** Data imported from outside the tap-tone-pi measurement pipeline.

**Examples:**
- Wood species reference data from databases
- Imported measurement bundles from other systems
- Literature values for material properties
- Calibration certificates from external labs

**Authority:** External authority — tap-tone-pi does not vouch for validity.

**Constraints:**
- Source must be identified
- Cannot claim OBSERVED status without re-measurement
- Downstream systems must preserve external-source marking

**Key distinction:** EXTERNALLY-SOURCED data carries whatever authority its source provides, which may be unknown or unverifiable.

---

### 3. Status Transitions

#### Allowed Transitions

```
OBSERVED → DERIVED (via deterministic transform)
OBSERVED → ESTIMATED (via inference)
DERIVED → ESTIMATED (via inference)
DERIVED → PREDICTED (via model)
ESTIMATED → PREDICTED (via model)
Any → HEURISTIC (via advisory interpretation)
Any → OPERATOR-ANNOTATED (via human judgment)
```

#### Forbidden Transitions

```
Any → OBSERVED (only physical capture creates OBSERVED)
HEURISTIC → Any higher authority (heuristic cannot elevate)
PREDICTED → OBSERVED (predictions cannot become measurements)
ESTIMATED → DERIVED (inference cannot become computation)
Any → canonical/validated/approved (forbidden states per ADR-0010)
```

### 4. Status Inheritance

**Rule:** Epistemic status does not elevate through combination.

When multiple sources are combined:
- The result takes the **lowest** authority status of any input
- Exception: OPERATOR-ANNOTATED additions do not downgrade, but are tracked separately

| Input A | Input B | Result |
|---------|---------|--------|
| OBSERVED | OBSERVED | DERIVED (combination is a transform) |
| OBSERVED | ESTIMATED | ESTIMATED |
| DERIVED | PREDICTED | PREDICTED |
| Any | HEURISTIC | HEURISTIC |
| Any | EXTERNALLY-SOURCED | EXTERNALLY-SOURCED (unless re-measured) |

### 5. Display Requirements

When epistemic status is displayed:

| Status | Display Indicator |
|--------|-------------------|
| OBSERVED | "Measured" or "Captured" |
| DERIVED | "Computed" or "Calculated" |
| ESTIMATED | "Estimated" or "Approximated" |
| PREDICTED | "Predicted" or "Modeled" |
| HEURISTIC | "Advisory" or "Suggested" |
| OPERATOR-ANNOTATED | "Operator note" or "Manual entry" |
| EXTERNALLY-SOURCED | "External" or "Imported" |

Display must NOT use:
- "Verified", "Validated", "Confirmed" for any status
- "Canonical", "Authoritative", "Official" for non-OBSERVED status
- Bare values without status context in mixed displays

## Relationship to Authority Laundering

The epistemic status taxonomy provides the constitutional mechanism to prevent authority laundering (as defined in ADR-0011).

At each transition point:
1. Status must be explicitly tracked
2. Status cannot be silently elevated
3. Accumulation of historical occurrences does not change status
4. Operator trust does not upgrade system status

```
OBSERVED (authoritative)
    │
    ▼ transform
DERIVED (computational)
    │
    ▼ inference
ESTIMATED (approximation)
    │
    ▼ advisory interpretation
HEURISTIC (no authority) ← CANNOT RETURN UPWARD
```

## Consequences

### Positive

- Complete classification of all artifact types
- Clear authority boundaries per category
- Transition rules prevent silent elevation
- Display requirements prevent presentation leakage
- Foundation for anti-laundering implementation

### Negative

- Requires tracking overhead in future implementations
- Some existing artifacts may need retroactive classification
- Display requirements may require UI updates
